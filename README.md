# Provenance Guard

A backend that a creative-sharing platform can plug in to estimate whether a
piece of text was written by a human or generated by AI. It returns a
confidence score, a plain-language transparency label, and lets creators appeal
a result. Every decision is written to a structured audit log.

The goal is **not** perfect AI detection (an unsolved problem). The goal is to
be honest about uncertainty and to never quietly accuse a human writer.

## Setup

```bash
python -m venv .venv
source .venv/bin/activate          # Windows: .venv\Scripts\activate
pip install -r requirements.txt
```

Create a `.env` file in the repo root (it is git-ignored):

```
GROQ_API_KEY=your_key_here
```

Run it:

```bash
python app.py        # serves on http://localhost:5000
```

## API

| Method | Route     | Body                              | Returns                                   |
|--------|-----------|-----------------------------------|-------------------------------------------|
| POST   | `/submit` | `text`, `creator_id`              | `content_id`, `attribution`, `confidence`, `label` |
| POST   | `/appeal` | `content_id`, `creator_reasoning` | confirmation + new `status`               |
| GET    | `/log`    | —                                 | recent audit-log entries                  |

Example:

```bash
curl -s -X POST http://localhost:5000/submit \
  -H "Content-Type: application/json" \
  -d '{"text": "The sun dipped below the horizon...", "creator_id": "test-user-1"}'
```

## Architecture

```
POST /submit (text, creator_id)
      |
      v
[Signal 1: Groq LLM] --llm_score--> +
      |                             |
      v                             v
[Signal 2: Stylometrics] --style_score--> [Confidence Scoring]
                                                |
                                                v combined_score
                                        [Transparency Label]
                                                |
                                                v label_text
                                          [Audit Log] --> response
```

Text is scored by two independent signals, blended into one confidence value,
mapped to a label, logged, and returned. An appeal looks the record up by
`content_id`, sets its status to `under_review`, and logs the creator's
reasoning next to the original decision. Full diagrams are in
[planning.md](planning.md).

## Detection Signals

The system uses two **distinct** signals — one semantic, one structural — so
they capture genuinely different properties of the text.

**Signal 1 — Groq LLM classification** (`signals.py: llm_signal`)
Sends the text to `llama-3.3-70b-versatile` and asks for an `ai_likelihood`
score from 0–1. This captures the *holistic feel* of the writing: does it read
as generic and evenly hedged (AI) or personal and irregular (human)?
*Why this signal:* it understands meaning and tone, which simple statistics
can't. *Blind spot:* it is non-deterministic, prompt-dependent, and tends to
over-flag formal or non-native human writing as AI.

**Signal 2 — Stylometric heuristics** (`signals.py: style_signal`)
Pure-Python statistics, no external libraries. Three metrics, each pointing
toward AI when the text looks uniform:
- **Sentence-length variation** (coefficient of variation) — AI keeps a steady
  rhythm; humans vary more.
- **Type-token ratio** (vocabulary diversity).
- **Punctuation density** (commas + semicolons per sentence) — AI tends to be
  evenly punctuated.
*Why this signal:* it measures structure the LLM ignores. *Blind spot:* it is
unreliable on short text and can't tell polished human writing from AI.

These are independent on purpose: one is about *meaning*, one about *form*. The
combination is more informative than either alone.

## Confidence Scoring

Both signals output a 0–1 AI-likelihood. They are combined with a weighted
average (`scoring.py`):

```
confidence = 0.6 * llm_score + 0.4 * style_score
```

The LLM is weighted higher because it reads meaning; stylometrics is a noisier
structural check. `confidence` is the AI-likelihood: **0.0 = clearly human,
1.0 = clearly AI**. A `0.6` is deliberately designed to land in the *uncertain*
band, so the system hedges instead of accusing.

**Thresholds (false-positive averse):**

| Confidence        | Attribution     |
|-------------------|-----------------|
| `>= 0.70`         | likely AI       |
| `0.40 – 0.70`     | uncertain       |
| `< 0.40`          | likely human    |

The "likely AI" bar is set high (0.70) on purpose: on a writing platform,
falsely calling a human's work AI is worse than missing an AI text, so we lean
toward *uncertain* when the signals aren't strongly in agreement.

**How I tested that scores are meaningful.** I ran four deliberately chosen
inputs (clearly AI, clearly human, and two borderline) and checked the scores
matched intuition, printing each signal separately when they didn't. Two real
examples from that testing:

| Input                | llm_score | style_score | **confidence** | label band   |
|----------------------|-----------|-------------|----------------|--------------|
| Polished AI essay    | 0.80      | 0.58        | **0.71**       | likely AI    |
| Casual ramen review  | 0.20      | 0.40        | **0.28**       | likely human |

The high-confidence and low-confidence cases are far apart (0.71 vs 0.28),
which shows the score varies meaningfully rather than hovering around a constant.

## Transparency Label

The label changes with the confidence band. Exact text of all three variants
(`{pct}` = `round(confidence * 100)`):

**High-confidence AI** (`confidence >= 0.70`):
> 🤖 This content is likely AI-generated. Our analysis found strong signals of
> automated writing (confidence: {pct}%). This is an automated estimate, not a
> certainty — the creator can appeal.

**High-confidence human** (`confidence < 0.40`):
> ✍️ This content appears to be human-written. We found no strong signs of AI
> generation (confidence it is human: {100-pct}%). This is an automated
> estimate, not a guarantee.

**Uncertain** (`0.40 <= confidence < 0.70`):
> ❓ We're not sure who wrote this. Our signals were mixed, so we can't
> confidently say whether it is AI-generated or human-written (AI-likelihood:
> {pct}%). Treat this result with caution.

Every label says "estimate, not certainty" and the AI one points to the appeal
path, because a false accusation is the worst outcome for a creator.

## Appeals Workflow

Any creator can contest a result by sending their `content_id` and
`creator_reasoning` to `POST /appeal`. The system:
1. looks up the original decision,
2. sets its status to `under_review`,
3. logs the appeal (reasoning + timestamp) next to the original scores,
4. returns a confirmation.

A human reviewer reading `GET /log` sees the full original record (scores,
label) plus the creator's reasoning and the `under_review` status. Automated
re-classification is intentionally not done — a human decides.

```bash
curl -s -X POST http://localhost:5000/appeal \
  -H "Content-Type: application/json" \
  -d '{"content_id": "PASTE-ID", "creator_reasoning": "I wrote this myself; I am a non-native English speaker."}'
```

## Rate Limiting

`POST /submit` is limited to **10 per minute; 100 per day** per IP
(Flask-Limiter, in-memory storage).

*Reasoning:* a real writer submits a handful of pieces — even an active one
rarely exceeds a few per minute — so 10/minute never gets in a human's way. But
it stops a script from flooding the (paid) LLM signal. The 100/day cap blocks
sustained abuse while staying well above any plausible single-writer day.

**Evidence** — 12 rapid requests (limit is 10/min):

The first ten succeed; the rest return `429 Too Many Requests`.

## Known Limitations

**Formal, academic, or non-native human writing** is the case this system most
likely gets wrong. In testing, a clean academic paragraph written by a human
scored `0.71` (likely AI). This isn't a tuning bug — both signals share the
blind spot: the LLM reads polished, evenly structured prose as "AI-like," and
the stylometrics see low sentence-length variation. Since both signals fail the
same way, no scoring tweak separates it from real AI text. The honest mitigation
is the **appeal workflow**, not pretending the detector is better than it is.

Two more weak spots: very short text (stylometrics needs enough words to be
meaningful) and lightly edited AI (it correctly lands mid-range / uncertain,
but that means it's never a confident catch).

## Spec Reflection

**Where the spec helped:** writing out the three label variants and the exact
thresholds *before* coding meant the scoring function and the label function had
a single shared contract. When I built the label generator in Milestone 5, there
was no guesswork — the bands were already decided in `planning.md`, so the code
just implemented them.

**Where I diverged:** my spec assumed borderline human writing would naturally
fall into the *uncertain* band. During calibration the LLM kept scoring formal
human text as confidently AI (0.8). I diverged by **adding explicit calibration
instructions to the LLM prompt** (use 0.4–0.6 when unsure; don't penalize
formality alone) — a step the spec didn't mention. It fixed the edited-AI case
but not the formal-human one, which I then accepted as a documented limitation.

## AI Usage

**1. Generating the scoring + signal scaffolding.** I gave the AI tool my
detection-signals and uncertainty sections plus the diagram and asked for the
stylometrics function and the combine logic. It produced a reasonable structure,
but I **overrode the type-token-ratio direction and the threshold mapping** so
they matched my spec's exact bands rather than the AI's slightly different ones.

**2. Recalibrating the LLM prompt.** When formal human text scored as AI, I
directed the AI tool to rewrite the detector system prompt to be
uncertainty-aware and false-positive averse. I kept its calibration wording but
**revised the score bands to match my thresholds** and verified the change by
re-running all four calibration inputs, not by trusting the output.
